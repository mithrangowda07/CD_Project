Compiler Testing Survey

CS

Csmith

random-c-program-generator · 2011 · Yang et al.

▾

Purpose: Generate random, valid C programs to stress-test C compilers and expose miscompilation bugs via differential testing.

How it works

  • Generates syntactically and semantically valid C programs from scratch using grammar-guided random production rules
  • Avoids undefined behaviour (UB) through careful pointer aliasing analysis and safe expression generation
  • Compiles each generated program with multiple compilers (GCC, Clang, ICC) at varying opt levels
  • Flags discrepancies in output as potential bugs — programs should produce identical results
  • Incorporates a runtime checker to catch crashes and sanitizer failures

Advantages

  • Produces standards-compliant C — maximises compiler portability testing
  • Has found hundreds of real bugs in GCC, Clang, and other production compilers
  • No seed corpus required — fully autonomous generation
  • Precise UB-avoidance lets results be interpreted unambiguously
  • Well-studied baseline; large body of follow-on work

Limitations

  • C-specific — cannot directly test LLVM IR passes or backend pipelines
  • Generated programs are structurally shallow; rarely stress advanced language features
  • UB-avoidance heuristics are conservative — large classes of legal programs are never explored
  • No feedback signal — mutation is blind, low efficiency vs coverage-guided tools
  • Slow: most generated programs are boring to the compiler's optimiser
YG

YARPGen

yet-another-random-program-generator · 2020 · Livinskii et al.

▾

Purpose: Generate C/C++ programs that specifically target and exercise compiler optimisation passes, improving on Csmith's ability to trigger optimiser bugs.

How it works

  • Uses a generate-and-filter approach: builds expression trees that are statically proven to be UB-free at generation time
  • Employs generation policies that bias toward patterns known to exercise specific optimisations (loop vectorisation, SLP, LICM, etc.)
  • Tracks live ranges and aliasing symbolically to prove safety without runtime checks
  • Outputs C/C++ source that is then compiled and differentially tested
  • Supports generating code that triggers specific LLVM opt passes explicitly

Advantages

  • Substantially better at triggering LLVM's middle-end optimiser bugs than Csmith
  • Symbolic UB-avoidance is faster and more complete than Csmith's runtime approach
  • Extensible policy system — new optimisation targets can be added
  • Found hundreds of new bugs not found by Csmith in LLVM/GCC
  • Supports C++ in addition to C, covering more of the compiler pipeline

Limitations

  • Still operates at C/C++ source level — IR passes are only indirectly tested
  • Policy design requires expert knowledge of which optimisations to target
  • No coverage feedback — generation remains structurally blind
  • Generated programs can be unrealistic / not representative of real-world code
  • Limited to arithmetic and loop-heavy code; poor coverage of memory-safety passes
CGF

Coverage-Guided Fuzzing

mutation-based · AFL / libFuzzer / honggfuzz paradigm

▾

Purpose: Automatically mutate inputs (LLVM IR, bitcode, or C source) guided by code coverage feedback to maximise path exploration in the compiler under test.

Workflow

  • Seed corpus: start with a set of known-good IR/bitcode files
  • Instrument target: compile the LLVM tool (e.g. opt, llc) with edge-coverage SanitizerCoverage
  • Mutate: apply byte-level mutations (bit flips, splices, insertions) to seeds
  • Execute: run mutated input through compiler, record edge coverage bitmap
  • Retain novelty: if new edges are discovered, add input to corpus
  • Report: crashes, hangs, and assertion failures are saved as bugs

Advantages

  • Works directly on LLVM IR bitcode — tests passes and backends that C-level tools miss
  • Coverage feedback makes exploration exponentially more efficient than blind mutation
  • Fully automated with minimal setup — pick a fuzzer and a corpus
  • libFuzzer natively integrated into LLVM's own testing infrastructure
  • Scales well: can run in parallel across many cores
  • Found critical bugs in clang's parser, LLVM backends, and sanitizer runtimes

Limitations

  • Byte-level mutations frequently produce syntactically invalid IR — rejected before reaching interesting passes
  • Hard to reach deep semantic bugs that require structurally valid inputs
  • Coverage saturation: after initial exploration, new interesting inputs become rare
  • Requires an instrumented build of the compiler — impractical for proprietary backends
  • Crash triage is manual and time-consuming; many bugs are duplicates
